## Methodology

- Public path: clean-installed built Python wheel → Go HTTP control plane → Agent Sandbox v0.5.2 → k3s → gVisor.
- Clock: monotonic `perf_counter_ns()`.
- Core series: 2 excluded warm-ups followed by 10 recorded samples.
- Concurrency series: 1 excluded warm-up followed by 3 recorded samples to keep the single-node run inside a practical time budget.
- Warm acquire measures only `SandboxClient.create()`; release and wait-for-replenishment are outside the timed region.
- Replenishment measures externally observable time from completed release request until `readyReplicas == desired replicas`.
- Chromium milestones use `performance.now()` inside the Sandbox and are not mixed with HTTP exec round-trip duration.
- Raw JSON was validated against `tests/benchmark/result.schema.v1.json`; all 22 series reported zero failures.

## Interpretation and limitations

- This is the first local baseline, not an SLO and not directly comparable to GKE capacity or latency.
- The environment has only 2 vCPU, 4 GiB RAM, one Kubernetes node, and one ready replica per Pool. Concurrency results therefore include cold Sandbox scheduling and resource contention after the first request.
- Percentiles from 10 samples, and especially 3 concurrency samples, are descriptive rather than statistically strong tail estimates.
- Cold image pulls, new-node startup, control-plane/controller resource telemetry, and sustained throughput remain open work in issue #1.
- File sizes reflect current effective protocol limits. Base64 expansion reaches the 1 MiB JSON request cap before a 1 MiB raw binary write, and reaches the 10 MiB command-output cap before a 10 MiB raw binary read. The baseline therefore uses 512 KiB writes and 7 MiB reads.
- Raw samples remain under the gitignored `.sandbox-platform/benchmarks/` directory and are not committed; this reviewed Markdown summary is the durable repository artifact.
